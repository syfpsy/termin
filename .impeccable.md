## Design Context

### Users

Phosphor is for motion designers and terminal-art enthusiasts who want to author CRT/terminal-native motion by describing intent, editing readable notation, and seeing an immediate live preview. The first working context is desktop authoring: a focused creative tool used for iterative scene design, not a casual mobile viewer.

The core job is to turn prose like "make a two second boot sequence, glitch, then reveal SYSTEM READY" into editable `.me` motion notation, preview it at a fixed tick rate, and let the user refine it without needing to code.

### Brand Personality

Precise, phosphorescent, workshop-grade.

The interface should feel like a serious motion instrument with a CRT soul: expressive and atmospheric in the preview, restrained and durable in the surrounding tool chrome. It must avoid becoming a generic retro terminal toy, a shader demo, or an AI chat app with decorative scanlines.

### Aesthetic Direction

Keep the design faithful to the Phosphor handoff: phosphor-on-black palette, light glow, discrete hardware-like motion, sharp panel geometry, VT323 only for display moments, and small mono UI type for long working sessions. The preview carries the theatrical CRT effect; the application shell stays utilitarian, dense, and legible.

Desktop is the primary layout target for the first release. Tablet and mobile should remain architecturally possible later, but must not compromise the preview-dominant workspace.

### Design Principles

1. Engine first, UI second: the product only works if Grid, PhosphorBuffer, Clock, DSL parsing, scene evaluation, and renderers stay cleanly separated.
2. Vibe coding is core, not an add-on: the director must be part of the first authoring loop, with provider boundaries that can support Claude, OpenAI, and local/mock generation.
3. The DSL must stay readable as stage directions: prose-adjacent lines, comments as intent, invalid lines tolerated, and preview rendering whatever parses.
4. The preview must feel terminal-native: fixed tick clock, discrete cells, phosphor decay, Canvas and WebGL renderers sharing the same buffer contract.
5. Future-proof by boundaries: local-first scene persistence, import/export of `.me`, renderer abstraction, provider abstraction, and export architecture that can later add MP4/GIF/WebM without reshaping the app.
